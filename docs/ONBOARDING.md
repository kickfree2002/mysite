# 新規参加者向けオンボーディング

## 1. コードベースの一般的な構造

現時点のリポジトリは非常に小さく、次のような構成です。

- `index.html`
  - ルートにあるエントリーポイントです。
  - ただし現在は **空ファイル**（0 bytes）で、実装は未着手です。
- `README.md`
  - プロジェクト概要と、このオンボーディング資料への導線です。
- `docs/ONBOARDING.md`
  - このガイドです。

> つまり、これは「機能実装済みアプリ」ではなく、これから育てるための初期スケルトンです。

## 2. 参加前に知っておく重要事項

### 技術的な前提

- ビルド設定（`package.json`, `vite.config.*`, `webpack.config.*` など）は未導入です。
- テスト基盤（Jest / Vitest / Playwright など）も未導入です。
- ディレクトリ規約（`src/`, `components/`, `tests/` など）は未確定です。

### 作業運用の前提

- まずは **アプリの最小要件を先に定義** することが重要です。
- いきなり実装を始めるより、以下を最初に揃えると手戻りを減らせます。
  1. 目的（何を作るか）
  2. 最小機能（MVP）
  3. 技術スタック（例: 素のHTML/CSS/JS か React/Vue か）
  4. 開発ルール（lint, format, テスト方針, ブランチ運用）

## 3. 次に学習・整備すると良い項目（推奨順）

1. **プロダクト要件の明文化**
   - まずは「誰が」「何のために」使うサイトかを1ページで整理。

2. **開発基盤の初期セットアップ**
   - 最小でも次のいずれかを導入。
     - A) 素のHTML/CSS/JSで開始（小規模向け）
     - B) Vite + フレームワークで開始（中長期向け）

3. **ディレクトリ標準化**
   - 例: `src/`, `public/`, `docs/`, `tests/`。
   - コードが増える前に決めるのがポイント。

4. **品質ゲートの導入**
   - Formatter（Prettier）
   - Linter（ESLint）
   - 最低限の自動テスト（スモークテスト）

5. **開発フローの定義**
   - PRテンプレート
   - Commitメッセージ規約
   - レビュー観点（UI, 可読性, テスト, 互換性）

## 4. 新規参加者の最初の1週間モデル

- Day 1: リポジトリ理解、要件の確認、ローカル起動方針の共有
- Day 2: 技術スタック決定、初期セットアップPR
- Day 3: ページ骨組み（ヘッダー/メイン/フッター）実装
- Day 4: スタイル基盤と共通コンポーネント導入
- Day 5: 最小機能の動作確認 + ドキュメント更新

## 5. 現時点の結論

このリポジトリは「実装理解」よりも「土台設計」が重要なフェーズです。
新規参加者は、まず機能開発より先に **方針の合意と開発基盤の整備** を優先してください。
